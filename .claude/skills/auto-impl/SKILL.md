# 実装をレビュー可能な状態まで完了させるスキル

**ゴール**: ユーザーがレビューするだけで済む状態 — テストが証明済みで、差分が明確で、PRが開いている — を自律的に作り上げること。

TDDはその品質保証の手段。ブランチ・コミット・プッシュ・PRは届け方の手段。

**前提条件**: `gh auth status` が成功していること（インストール済み＋GitHubアカウントに認証済み）。満たされていない場合はユーザーに確認を促して止まること。

---

## Phase 1: TDD スキルで実装の品質を保証する

**`tdd` スキルを呼び出して実行する。** TDDの具体的な手順（規約確認・Red確認・Green確認）はそちらに委ねる。

`tdd` スキルが完了し、テストスイート全件パスした状態になってから Phase 2 に進む。

---

## Phase 2: レビュー可能な状態に届ける

実装の品質が保証されたら、レビュアーが差分を追いやすい形で届ける。

### ブランチを切る

変更の意図が伝わるブランチ名をつける:

- バグ修正 → `fix/<内容>`（例: `fix/audio-generation-timeout-handling`）
- 新機能 → `feature/<内容>`
- リファクタ → `refactor/<内容>`

英語・ケバブケース。何の変更かが名前だけで伝わること。

```bash
git checkout -b fix/<内容>
```

### コミットする

変更したファイルだけをステージする（`git add -A` は使わない）。

コミットメッセージは日本語で「何をなぜ変えたか」を書く:

```
fix: 音声生成タイムアウト時にエラーをログ出力してワークフローを継続する

wait_for_completion の TimeoutError をキャッチし WARNING ログを出す。
ノートブック作成は成功しているため再 raise しない。

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
```

末尾に必ず `Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>` をつける。

### プッシュして PR を作る

```bash
git push -u origin <ブランチ名>
gh pr create --title "<タイトル>" --body "..."
```

PR 本文には以下を含める:
- **Summary**: 何をなぜ変えたか（箇条書き3点まで）
- **Test plan**: レビュアーが確認すべき点のチェックリスト

作成後、PR の URL をユーザーに伝えて完了。

---

## 完了の定義

- [ ] `tdd` スキルが完了し、テストスイート全件パスしている
- [ ] ブランチが切られている
- [ ] コミットメッセージが変更の意図を伝えている
- [ ] PR が開いていて URL を共有済み

これがすべて揃ったとき、ユーザーはレビューするだけでよい状態になっている。
